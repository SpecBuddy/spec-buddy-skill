# SpecBuddy — Концепция инструмента

---

## 1. Общее описание

### 1.1. Назначение: разработка с ИИ

**SpecBuddy** — инструмент, предназначенный для **разработки с ИИ**. Чтобы агент мог осмысленно выполнять задачи, нужны два основания:

1. **Спека** — что именно нужно сделать (цель, план, шаги, критерии).
2. **Тулчейн** — какими инструментами агент пользуется (файлы, репозиторий, MCP-серверы, API и т.д.).

SpecBuddy даёт и то и другое: и среду для написания/ведения спеки, и способ описать и подключить тулчейн (в том числе через MCP).

### 1.2. Разработка от спецификации

Мы не только «разрабатываем от спецификации», но делаем **инструмент под разработку с ИИ**; спека и тулчейн — необходимое для этого условие. Исходная точка — спека: в ней описано, что делать. Инструмент позволяет спеку писать, уточнять и выполнять по шагам, передавая шаги агенту, который действует в рамках заданного тулчейна.

Разработка ведётся от спецификации к коду: сначала формализуются «что делать» и «чем пользоваться», затем шаги плана выполняются агентом с возможностью корректировки плана и самой спеки.

## 2. Спека и тулчейн

### 2.1. Возможности по написанию спеки

- **AI-driven** — генерация и доработка спеки с помощью агента (из промпта, исследование, уточнение шагов).
- **Детерминированные** — упрощённый ввод и привязка сущностей без участия модели:
  - ссылки на файлы;
  - пакеты, символы в коде;
  - текстовые диапазоны (ranges);
  - MCP-тулы (часть тулчейна);
  - ссылки на другие спеки.

### 2.2. Формат спеки (минимальное описание)

Спека — **обычный Markdown-файл**. Без YAML-блоков, своих fenced-структур и кастомных аннотаций; структура задаётся естественной разметкой Markdown (заголовки, списки, нумерация).

**Inline-маркеры** позволяют привязать контекст и инструменты прямо в тексте спеки:
- `@file:path/to/file` — файл целиком в контекст агента;
- `@range:path/to/file#L10-L42` — только указанный диапазон строк;
- `@cmd:./gradlew test` — команда оболочки (должна быть в allowlist);
- `@mcp:tool.name` или `@mcp:tool.name(param="value")` — вызов MCP-тула;
- `@spec:path/to/other-spec.md` — вставка другой спеки (композиция).

**Шаги** выделяются стандартными средствами Markdown — заголовками или нумерованными пунктами. SpecBuddy разбивает спеку на шаги по структуре документа: у каждого шага есть заголовок/номер, тело (Markdown), входы (все `@file`, `@range`, `@spec`) и инструменты (все `@mcp`, `@cmd`). Дополнительной схемы не требуется — SpecBuddy по структуре документа и маркерам строит граф выполнения.

*Полное описание формата и модели выполнения — в документе «SpecBuddy — Product & Technical Specification».*

### 2.3. Тулчейн: шаблоны и примеры спек

Тулчейн определяет, **чем агент может пользоваться** при выполнении шагов (поиск по коду, вызов API, базы, браузер и т.д.). Часть тулчейна — **MCP-инструменты**: они описываются в среде спеки (или в конфигурации), чтобы агент знал, какие тулы доступны и как их вызывать.

**Что даём для тулчейнов:** для разных сценариев SpecBuddy предоставляет **шаблоны тулчейнов** (в т.ч. шаблоны под популярные MCP-серверы) и **примеры спек под эти шаблоны**. Шаблон тулчейна — готовый набор инструментов (например, файловая система + GitHub + база знаний). Пример спек — образец того, как писать спеку, рассчитанную на этот тулчейн: какие шаги, какие ссылки на файлы/символы, как вызывать тулы. Так пользователь быстрее стартует: выбирает шаблон тулчейна и при необходимости берёт за основу пример спеки.

## 3. Создание и уточнение спеки, выполнение с агентом

Шаги плана выполняются агентом в окружении с заданным тулчейном (шаблон тулчейна можно взять из набора шаблонов и при необходимости опереться на пример спеки для него).

### 3.1. Создание спеки из текстового описания

1. Пользователь вводит **простой промпт** (краткое описание желаемого).
2. Инструмент запускает **агента**, который превращает промпт в **более подробную спеку**.
3. При необходимости запускается **исследование**: имея часть спеки, агент уточняет и дополняет её (по коду, документации, другим спекам).

### 3.2. План и шаги

- В спеке может быть описан **план**, разбитый на **шаги**.
- Шаги можно **выполнять по очереди**, передавая их агенту.
- **После выполнения шага** агент может:
  - корректировать **следующие шаги** с учётом сделанного;
  - вносить **правки в текущий шаг** (например, уточнение формулировки или критериев готовности).
- Есть **инструментарий по перегруппировке шагов** (объединение, разбиение, изменение порядка).

## 4. Сводка возможностей

| Область | Возможности |
|--------|-------------|
| **Назначение** | Инструмент для разработки с ИИ; основа — спека (что делать) + тулчейн (чем пользоваться). |
| Написание спеки | AI: из промпта, исследование, уточнение. Детерминировано: файлы, пакеты, символы, ranges, MCP, другие спеки. |
| Тулчейн | Шаблоны тулчейнов (в т.ч. под популярные MCP) + примеры спек под каждый шаблон — чтобы быстро стартовать с нужным набором инструментов и образцом спеки. |
| План | План в спеке, разбиение на шаги; выполнение в рамках заданного тулчейна. |
| Выполнение | Пошаговая передача шагов агенту, пост-коррекция шагов и плана. |
| Структура плана | Перегруппировка, объединение и разбиение шагов. |

## 5. Открытые вопросы (для дальнейшей проработки)

- Формат хранения спеки (Markdown, YAML, свой DSL, привязка к репозиторию).
- Формат описания тулчейна и привязка MCP к спеке/проекту.
- Интеграция с IDE / Cursor / CLI.
- Версионирование спеки и связь с историей изменений кода.
- Критерии готовности шага и проверка выполнения агентом.
- Повторное использование спек (композиция, шаблоны, библиотека спек).
